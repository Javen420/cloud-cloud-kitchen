── Composite Services ───────────────────────────────────
Use kebab case for folder name (e.g. assign-driver)

── What composite services do ───────────────────────────
- Orchestrate multiple  services into a single workflow
- Call  services via HTTP or publish to RabbitMQ
- Do NOT contain core business logic — that belongs in  services

── Your routes and ports (from kong.yml) ────────────────
Your endpoint paths and ports MUST match these exactly.

Scenario 1:
  order-fulfilment       | port 8081 | POST /api/v1/order/submit
                                      | GET  /api/v1/order/{order_id}
  payment (atomic)       | port 8089 | POST, GET /api/v1/payment/*
  menu (external)        | via Kong  | GET /api/v1/menu

Scenario 2:
  assign-kitchen         | port 8093 | background poller only; GET /health (not exposed via Kong)
  coordinate-fulfilment  | port 8094 | GET /orders, PUT /orders/{id}/status (kitchen dashboard)
  Kong kitchen gateway   | port 8000 | /api/v1/kitchen/* → coordinate-fulfilment

UI gateway pattern (match OrderUI/RiderUI/KitchenUI):
  BASE_URL = import.meta.env.VITE_API_BASE_URL || ""
  Use fetch(`${BASE_URL}/api/v1/...`) from UI clients
  Local Vite may proxy /api to Kong; in Docker/browser, BASE_URL can be http://localhost:8000

Scenario 3:
  assign-driver          | port 8086 | GET, PUT /api/v1/driver/assign
                                      | GET      /api/v1/driver/orders
  eta-tracking           | port 8087 | GET, POST /api/v1/eta/*
  rider UI               | port 5175 | uses Kong route base /api/v1/driver/* and /api/v1/eta/*

── Order-fulfilment flow (Scenario 1) ─────────────────
The order flow is fully synchronous (no Stripe redirects/webhooks):
  1. UI calls POST /api/v1/order/submit → order-fulfilment (8081)
  2. order-fulfilment calculates total from items
  3. Calls payment service POST http://payment:8089/api/v1/payment
  4. If payment fails → return error to UI
  5. Calls new-orders POST http://new-orders:8082/api/v1/orders
  6. Publishes AMQP notification (exchange: order_events, key: order.created)
  7. Returns confirmation to UI

── Calling other services ───────────────────────────────
- Use the Docker service name as hostname:
    requests.get("http://new-orders:8082/api/v1/orders?status=pending")
- For async, publish to RabbitMQ:
    RabbitMQ URL: os.getenv("RABBITMQ_URL")

── Adding to Docker ─────────────────────────────────────
Add your service to docker-compose.yml under the correct scenario comment:

  order-fulfilment:
    build: ../Composite Services/order-fulfilment
    env_file: ../.env
    ports:
      - "8081:8081"
    depends_on:
      rabbitmq:
        condition: service_healthy
